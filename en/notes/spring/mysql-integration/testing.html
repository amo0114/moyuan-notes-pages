第 9 章 · StockService 重构与事务边界 ​

9.1 目标 ​

把原本基于 HashMap 的 StockService 改成：

  • 入口校验参数
  • 委托给策略层做扣减
  • 同事务写流水
  • 乐观锁冲突时，用独立 Bean 的 REQUIRES_NEW 写一条失败流水，然后 rethrow
  • 整体行为对 StockController 零侵入（仍返回 Integer：null/−1/正数）

9.2 概念：@Transactional 是怎么工作的 ​

Spring 看到 @Transactional 后，给这个 Bean 生成一个代理。调用方拿到的不是 Bean 本身，而是代理对象：

caller ──▶ proxy(StockService) ──▶ 开事务 ──▶ 真 StockService.deductStock() ──▶ 提交事务

代理在调用前后插入"开事务 / 提交事务 / 回滚事务"的逻辑。

9.3 致命陷阱：self-invocation ​

如果在同一个类里方法 A 调用方法 B，B 标了 @Transactional，A 没标：

java
public void methodA() {
    methodB();   // ← 直接调用, 不走代理, @Transactional 不生效!
}

@Transactional
public void methodB() { ... }

this.methodB() 是直接调用真对象，绕开了代理。自调用永远绕开 AOP。

这就是为什么我们的冲突审计要放在独立 Bean 里：

java
StockService.deductStock() {
    try {
        ...业务...
    } catch (OptimisticLockConflictException e) {
        auditService.writeConflictLog(...);   // ← 调用的是另一个 Bean, 走代理, REQUIRES_NEW 生效
        throw e;
    }
}

9.4 概念：REQUIRES_NEW 跟主事务的关系 ​

主事务 (StockService.deductStock, PROPAGATION=REQUIRED)
   ├─ INSERT into stock_deduct_log ...    ← 成功路径
   ├─ UPDATE stock_item ...               ← 业务变更
   ├─ throw OptimisticLockConflictException
   │
   ├─ auditService.writeConflictLog()     ← 这里嵌套调用
   │     └─ REQUIRES_NEW 创建独立事务
   │         ├─ INSERT into stock_deduct_log (result_code=409, ...)
   │         └─ COMMIT  ← 独立事务先提交!
   │
   └─ 主事务 ROLLBACK (因为异常抛上去)
       ↑ 这里只回滚自己, 不影响已提交的 REQUIRES_NEW 事务

效果：库存没变（主事务回滚），但流水里多了一条 409 失败记录（独立事务已提交）。审计完整性得以保留。

9.5 编写 StockDeductAuditService ​

路径：stock-service/src/main/java/com/snowvictor/cloudwork/stock/service/StockDeductAuditService.java

java
package com.snowvictor.cloudwork.stock.service;

import com.snowvictor.cloudwork.stock.domain.StockDeductLog;
import com.snowvictor.cloudwork.stock.mapper.StockDeductLogMapper;
import com.snowvictor.cloudwork.stock.service.deduct.LockMode;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Propagation;
import org.springframework.transaction.annotation.Transactional;

/**
 * 独立 Bean: 仅承担乐观锁冲突的失败流水写入。
 * 必须独立于 StockService, 否则 Spring AOP self-invocation 会让 REQUIRES_NEW 失效。
 */
@Service
@RequiredArgsConstructor
@Slf4j
public class StockDeductAuditService {

    private final StockDeductLogMapper logMapper;

    @Transactional(propagation = Propagation.REQUIRES_NEW, rollbackFor = Exception.class)
    public void writeConflictLog(String traceId, String productId, int quantity, LockMode mode) {
        StockDeductLog log = new StockDeductLog();
        log.setTraceId(traceId);
        log.setProductId(productId);
        log.setQuantity(quantity);
        log.setResultCode(409);
        log.setRemainingStock(null);
        log.setLockMode(mode.name());
        log.setCostMs(0);
        logMapper.insert(log);
    }
}

💡 rollbackFor = Exception.class：默认 @Transactional 只对 RuntimeException 回滚。显式加上 rollbackFor = Exception.class 让 checked exception 也回滚。这是个常见最佳实践。

9.6 重构 StockService ​

整体替换 stock-service/src/main/java/com/snowvictor/cloudwork/stock/service/StockService.java：

java
package com.snowvictor.cloudwork.stock.service;

import com.snowvictor.cloudwork.stock.domain.StockDeductLog;
import com.snowvictor.cloudwork.stock.exception.InvalidDeductRequestException;
import com.snowvictor.cloudwork.stock.exception.OptimisticLockConflictException;
import com.snowvictor.cloudwork.stock.mapper.StockDeductLogMapper;
import com.snowvictor.cloudwork.stock.service.deduct.DeductOutcome;
import com.snowvictor.cloudwork.stock.service.deduct.StockDeductStrategy;
import com.snowvictor.cloudwork.stock.service.deduct.StockDeductStrategyResolver;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

@Service
@RequiredArgsConstructor
@Slf4j
public class StockService {

    private final StockDeductStrategyResolver resolver;
    private final StockDeductLogMapper logMapper;
    private final StockDeductAuditService auditService;   // 独立 Bean, 避免 self-invocation

    /**
     * 扣减库存。HTTP 契约（与原 StockController 配合）保持不变：
     *   return null  → 商品不存在 (StockController 转 HTTP 404)
     *   return -1    → 库存不足   (StockController 转 HTTP 400)
     *   return >= 0  → 扣减成功, 值为剩余库存
     * 冲突 / 非法参数 / 配置错误均通过异常抛出，由 StockServiceExceptionHandler 处理。
     */
    @Transactional(rollbackFor = Exception.class)
    public Integer deductStock(String productId, Integer qty, String traceId) {
        // 1) 入口校验, 不进入策略层, 不写流水
        if (qty == null || qty <= 0) {
            throw new InvalidDeductRequestException("quantity must be positive: " + qty);
        }
        // 2) 解析策略 (Nacos 配错时抛 UnsupportedLockModeException)
        StockDeductStrategy strategy = resolver.resolve();
        long start = System.currentTimeMillis();
        try {
            DeductOutcome outcome = strategy.deduct(productId, qty);
            // 3) 主事务同事务写流水 (success / insufficient / notFound 都走这里)
            logMapper.insert(toLog(traceId, productId, qty, outcome,
                                   strategy.mode().name(),
                                   System.currentTimeMillis() - start));
            return toLegacyReturn(outcome);
        } catch (OptimisticLockConflictException e) {
            // 4) 仅冲突场景: 主事务回滚, 独立 Bean 用 REQUIRES_NEW 写失败流水
            auditService.writeConflictLog(traceId, productId, qty, strategy.mode());
            throw e;
        }
    }

    private StockDeductLog toLog(String traceId, String productId, int qty,
                                 DeductOutcome out, String mode, long costMs) {
        StockDeductLog log = new StockDeductLog();
        log.setTraceId(traceId);
        log.setProductId(productId);
        log.setQuantity(qty);
        log.setResultCode(out.resultCode());
        log.setRemainingStock(out.remainingStock());
        log.setLockMode(mode);
        log.setCostMs((int) costMs);
        return log;
    }

    /** 翻译 DeductOutcome 回旧 StockController 期望的 Integer 三态。 */
    private Integer toLegacyReturn(DeductOutcome outcome) {
        return switch (outcome.resultCode()) {
            case 200 -> outcome.remainingStock();
            case -1  -> -1;
            case 404 -> null;
            default  -> throw new IllegalStateException(
                    "unexpected outcome: " + outcome.resultCode());
        };
    }
}

4 段逻辑：

  1. 入口校验：qty<=0 直接抛 InvalidDeductRequestException。不进入事务、不写流水。理由：脏数据不应该污染审计表
  2. 策略解析：Nacos 配错抛 UnsupportedLockModeException
  3. 主事务执行：扣减 + 同事务写流水。成功/不足/不存在三种 outcome 都走这里
  4. 冲突分支：策略抛 OptimisticLockConflictException 时，调独立 Bean 写 409 流水（REQUIRES_NEW 独立提交），然后 rethrow

💡 为什么 toLegacyReturn 把 DeductOutcome 翻译回 Integer？因为现有 StockController 是基于 null/-1/正数三态判断 HTTP 状态码的。我们承诺过"对 StockController 零侵入"，所以保留这个翻译层。

import org.junit.jupiter.api.extension.ExtendWith;
import org.mockito.InjectMocks;

import static org.assertj.core.api.Assertions.assertThat;
import static org.assertj.core.api.Assertions.assertThatThrownBy;
import static org.mockito.ArgumentMatchers.any;
import static org.mockito.ArgumentMatchers.anyInt;

import static org.mockito.Mockito.verify;

@ExtendWith(MockitoExtension.class)
class StockServiceTest {

    @Mock StockDeductStrategyResolver resolver;
    @Mock StockDeductStrategy strategy;
    @Mock StockDeductLogMapper logMapper;
    @Mock StockDeductAuditService auditService;
    @InjectMocks StockService stockService;

    @Test
    void rejects_nullOrNonPositiveQty() {
        assertThatThrownBy(() -> stockService.deductStock("p1001", null, "t1"))
                .isInstanceOf(InvalidDeductRequestException.class);
        assertThatThrownBy(() -> stockService.deductStock("p1001", 0, "t1"))
                .isInstanceOf(InvalidDeductRequestException.class);
        assertThatThrownBy(() -> stockService.deductStock("p1001", -1, "t1"))
                .isInstanceOf(InvalidDeductRequestException.class);
        verify(resolver, never()).resolve();
        verify(logMapper, never()).insert(any());
    }

    @Test
    void success_returnsRemainingAndWritesLog() {
        when(resolver.resolve()).thenReturn(strategy);
        when(strategy.mode()).thenReturn(LockMode.OPTIMISTIC);
        when(strategy.deduct("p1001", 3)).thenReturn(DeductOutcome.success(7));

        Integer remaining = stockService.deductStock("p1001", 3, "t1");
        assertThat(remaining).isEqualTo(7);
        verify(logMapper, times(1)).insert(any(StockDeductLog.class));
        verify(auditService, never()).writeConflictLog(any(), any(), anyInt(), any());
    }

    @Test
    void notFound_returnsNull() {
        when(resolver.resolve()).thenReturn(strategy);
        when(strategy.mode()).thenReturn(LockMode.OPTIMISTIC);
        when(strategy.deduct("p1001", 3)).thenReturn(DeductOutcome.notFound());

        assertThat(stockService.deductStock("p1001", 3, "t1")).isNull();
        verify(logMapper).insert(any(StockDeductLog.class));
    }

    @Test
    void insufficient_returnsMinusOne() {
        when(resolver.resolve()).thenReturn(strategy);
        when(strategy.mode()).thenReturn(LockMode.OPTIMISTIC);
        when(strategy.deduct("p1001", 999)).thenReturn(DeductOutcome.insufficient());

        assertThat(stockService.deductStock("p1001", 999, "t1")).isEqualTo(-1);
    }

    @Test
    void conflict_invokesAuditAndRethrows() {
        when(resolver.resolve()).thenReturn(strategy);
        when(strategy.mode()).thenReturn(LockMode.OPTIMISTIC);
        when(strategy.deduct("p1001", 3))
                .thenThrow(new OptimisticLockConflictException("boom"));

        assertThatThrownBy(() -> stockService.deductStock("p1001", 3, "t1"))
                .isInstanceOf(OptimisticLockConflictException.class);
        verify(auditService).writeConflictLog("t1", "p1001", 3, LockMode.OPTIMISTIC);
    }
}

第 10 章 · 全局异常处理器 ​

10.1 目标 ​

把第 6 章定义的 3 个领域异常映射成有意义的 HTTP 状态码 + JSON 响应。

异常HTTP 状态码
InvalidDeductRequestException400 Bad Request
OptimisticLockConflictException409 Conflict
UnsupportedLockModeException500 Internal Server Error

10.2 概念：@RestControllerAdvice ​

Spring 的全局异常拦截器。任何 @RestController 抛的异常，都会先经过它。匹配上 @ExceptionHandler 注解的方法就用那个方法处理。

10.3 完整实现 ​

路径：stock-service/src/main/java/com/snowvictor/cloudwork/stock/handler/StockServiceExceptionHandler.java

java
package com.snowvictor.cloudwork.stock.handler;

import com.snowvictor.cloudwork.stock.exception.InvalidDeductRequestException;
import com.snowvictor.cloudwork.stock.exception.OptimisticLockConflictException;
import com.snowvictor.cloudwork.stock.exception.UnsupportedLockModeException;
import lombok.extern.slf4j.Slf4j;
import org.springframework.http.HttpStatus;
import org.springframework.http.ResponseEntity;
import org.springframework.web.bind.annotation.ExceptionHandler;
import org.springframework.web.bind.annotation.RestControllerAdvice;

import java.util.LinkedHashMap;
import java.util.Map;

@RestControllerAdvice
@Slf4j
public class StockServiceExceptionHandler {

    @ExceptionHandler(InvalidDeductRequestException.class)
    public ResponseEntity<Map<String, Object>> handleInvalid(InvalidDeductRequestException e) {
        log.warn("invalid deduct request: {}", e.getMessage());
        return error(HttpStatus.BAD_REQUEST, e.getMessage());
    }

    @ExceptionHandler(OptimisticLockConflictException.class)
    public ResponseEntity<Map<String, Object>> handleConflict(OptimisticLockConflictException e) {
        log.warn("optimistic lock conflict: {}", e.getMessage());
        return error(HttpStatus.CONFLICT, e.getMessage());
    }

    @ExceptionHandler(UnsupportedLockModeException.class)
    public ResponseEntity<Map<String, Object>> handleUnsupported(UnsupportedLockModeException e) {
        log.error("unsupported lock mode: {}", e.getMessage());
        return error(HttpStatus.INTERNAL_SERVER_ERROR, e.getMessage());
    }

    private ResponseEntity<Map<String, Object>> error(HttpStatus status, String message) {
        Map<String, Object> body = new LinkedHashMap<>();
        body.put("code", status.value());
        body.put("message", message);
        body.put("data", null);
        return ResponseEntity.status(status).body(body);
    }
}

💡 LinkedHashMap 保证字段顺序。如果用普通 HashMap，序列化出来的 JSON 字段顺序不稳定，前端解析时偶尔会"看起来不一样"造成困扰。

⚠️ gateway-service 的 GlobalExceptionHandler 不在这里。那是 WebFlux 反应式栈的处理器（gateway 用 Spring Cloud Gateway）；stock-service 是传统 Spring MVC 栈，要自己写。两个不能混用。

可以写测试，但收益不大。这个类基本上是"接收异常，包装成响应体"的纯粹翻译层，逻辑非常简单。集成测试（第 12 章）会自然覆盖到。

第 11 章 · order-service 改造与跨服务事务边界 ​

11.1 目标 ​

  • 删掉 OrderService.orderStore 这个 HashMap
  • 改为通过 OrderInfoMapper.insert 持久化
  • @Transactional 仅覆盖 insert，不包裹 Feign 调用

11.2 概念：为什么 Feign 不能在事务里 ​

java
// ❌ 错误示范
@Transactional
public OrderCreateResult createOrder(...) {
    StockDeductResponse resp = stockClient.deductStock(...);   // ← Feign 调用
    if (resp.getCode() == 200) {
        orderMapper.insert(orderInfo);
    }
}

问题：

  1. 事务长：Feign 调用要走 HTTP + 网络往返，可能 50ms-500ms。这段时间事务一直开着，连接池连接被占用
  2. 数据库锁延长：如果 insert 之前就开了事务，业务变更里如果还有更新别的表，那些表的行锁也会被多持有这么久
  3. 资源耗尽：高并发时大量连接被网络 IO 卡住，连接池耗尽

正确做法：Feign 在事务外，事务尽量薄。

11.3 改造后的 OrderService（含独立 OrderPersistenceService） ​

本节需要新建一个 OrderPersistenceService（承载 @Transactional），并整体替换 OrderService.java（只做编排，不带事务）。

第 1 个文件：新建 OrderPersistenceService.java

路径：order-service/src/main/java/com/snowvictor/cloudwork/order/service/OrderPersistenceService.java

java
package com.snowvictor.cloudwork.order.service;

import com.snowvictor.cloudwork.api.stock.dto.StockDeductResponse;
import com.snowvictor.cloudwork.order.domain.OrderInfo;
import com.snowvictor.cloudwork.order.dto.OrderCreateResult;
import com.snowvictor.cloudwork.order.mapper.OrderInfoMapper;
import lombok.RequiredArgsConstructor;
import org.springframework.stereotype.Service;
import org.springframework.transaction.annotation.Transactional;

/**
 * 独立 Bean: 仅承担订单的本地事务持久化。
 * 必须独立于 OrderService —— 如果写成 OrderService 的私有方法,
 * createOrder() 直接调用就是 self-invocation, Spring AOP 代理被绕开,
 * @Transactional 不会生效。详见教程第 9.3 节。
 *
 * 事务边界: 仅覆盖 insert, 不包含上游的 Feign 调用 (避免长事务持锁)。
 */
@Service
@RequiredArgsConstructor
public class OrderPersistenceService {

    private final OrderInfoMapper orderMapper;

    @Transactional(rollbackFor = Exception.class)
    public OrderCreateResult persistAndReturn(OrderInfo info, StockDeductResponse resp) {
        orderMapper.insert(info);
        return new OrderCreateResult(
                info.getOrderId(),
                info.getStatus(),
                info.getUserId(),
                info.getUserName(),
                info.getProductId(),
                info.getProductName(),
                info.getQuantity(),
                info.getPrice(),
                info.getTotalAmount(),
                info.getFallbackMessage(),
                resp.getData()
        );
    }
}

第 2 个文件：整体替换 OrderService.java

java
package com.snowvictor.cloudwork.order.service;

import com.snowvictor.cloudwork.api.stock.dto.StockDeductRequest;
import com.snowvictor.cloudwork.api.stock.dto.StockDeductResponse;
import com.snowvictor.cloudwork.order.client.StockClient;
import com.snowvictor.cloudwork.order.domain.OrderInfo;
import com.snowvictor.cloudwork.order.dto.OrderCreateResult;
import com.snowvictor.cloudwork.order.dto.OrderRequest;
import lombok.RequiredArgsConstructor;
import lombok.extern.slf4j.Slf4j;
import org.springframework.stereotype.Service;

import java.math.BigDecimal;
import java.time.LocalDateTime;
import java.time.format.DateTimeFormatter;
import java.util.UUID;

@Service
@Slf4j
@RequiredArgsConstructor
public class OrderService {

    private final StockClient stockClient;
    private final OrderPersistenceService orderPersistenceService;   // 独立 Bean, 触发 Spring 代理, 让 @Transactional 真正生效

    public OrderCreateResult createOrder(
            OrderRequest request, String userId, String userName, String userRole, String traceId
    ) {
        log.info("[{}] 开始创建订单: userId={}, productId={}, quantity={}",
                traceId, userId, request.getProductId(), request.getQuantity());

        StockDeductRequest stockRequest = new StockDeductRequest();
        stockRequest.setProductId(request.getProductId());
        stockRequest.setQuantity(request.getQuantity());

        // ─── Feign 调用在事务外 ─────────────────────────────────────────
        StockDeductResponse stockDeductResponse = stockClient.deductStock(
                userId, userName, userRole, traceId, stockRequest);

        if (stockDeductResponse == null) {
            log.error("[{}] 库存服务无响应", traceId);
            throw new IllegalStateException("库存服务无响应");
        }

        String orderId = generateOrderId();
        BigDecimal totalAmount = request.getPrice()
                .multiply(BigDecimal.valueOf(request.getQuantity()));

        if (stockDeductResponse.getCode() == 200) {
            Integer remaining = stockDeductResponse.getData() != null
                    ? stockDeductResponse.getData().getRemainingStock() : null;
            OrderInfo info = buildOrderInfo(orderId, "CREATED", userId, userName,
                    request, totalAmount, remaining, null, traceId);
            // 通过独立 Bean 触发代理, @Transactional 生效
            return orderPersistenceService.persistAndReturn(info, stockDeductResponse);
        }
        if (stockDeductResponse.getCode() == 503) {
            String fallbackMessage = "库存服务暂不可用，订单已降级创建，请后续人工核对库存";
            OrderInfo info = buildOrderInfo(orderId, "CREATED_FALLBACK", userId, userName,
                    request, totalAmount, null, fallbackMessage, traceId);
            return orderPersistenceService.persistAndReturn(info, stockDeductResponse);
        }
        throw new IllegalArgumentException(stockDeductResponse.getMessage());
    }

    private OrderInfo buildOrderInfo(String orderId, String status, String userId, String userName,
                                     OrderRequest req, BigDecimal totalAmount,
                                     Integer remainStock, String fallbackMessage, String traceId) {
        OrderInfo info = new OrderInfo();
        info.setOrderId(orderId);
        info.setStatus(status);
        info.setUserId(userId);
        info.setUserName(userName);
        info.setProductId(req.getProductId());
        info.setProductName(req.getProductName());
        info.setQuantity(req.getQuantity());
        info.setPrice(req.getPrice());
        info.setTotalAmount(totalAmount);
        info.setRemainStock(remainStock);
        info.setFallbackMessage(fallbackMessage);
        info.setTraceId(traceId);
        return info;
    }

    private String generateOrderId() {
        return "ORD" +
                LocalDateTime.now().format(DateTimeFormatter.ofPattern("yyyyMMddHHmmss")) +
                UUID.randomUUID().toString().substring(0, 4).toUpperCase();
    }
}

💡 为什么必须拆出 OrderPersistenceService？ 第 9.3 节讲过 self-invocation 陷阱：同类内方法 A 直接调方法 B（B 上有 @Transactional），调用走的是真实对象而非代理，注解失效。如果把 persistAndReturn 写成 OrderService 的 protected 方法，createOrder 调用它就是 self-invocation —— @Transactional 不启用，订单 insert 失败时不会回滚。和第 9 章 StockDeductAuditService 同源约束：但凡靠 @Transactional 触发代理的方法，都必须放在调用方之外的 Bean 上。这是本教程唯一一处与"文件最少化"原则冲突的地方，但事务正确性优先级更高。

11.4 跨服务事务的诚实声明 ​

仔细想：如果 Feign 已经成功扣了库存，但本地 orderMapper.insert 失败了，怎么办？

  • 库存被扣了，但订单没有创建
  • 数据不一致

我们显式选择不处理。理由：

  • 真正的解决方案是"本地消息表 / Outbox 模式"（见设计文档 §11 B4）或"Seata"（B5），都需要大量基础设施
  • 本次范围有限，写到这里已经很多东西要学了
  • 故意让错误向上抛，让 caller 知道有问题，作为下一迭代的伏笔

这是一个清晰的工程决策：先识别问题，承认问题，在范围内不解决，但记录到 Future Work。学习项目里这种取舍能力比技术细节更值钱。

第 12 章 · 用 Testcontainers 写真实集成测试 ​

12.1 目标 ​

  • 在测试时启动一个真实的 MySQL 容器
  • 让 Spring Boot 连这个容器，跑 Flyway，建表，做实际 DB 操作
  • 用并发 100 线程压测，断言零超卖

12.2 为什么不用 H2 ​

H2 是个 Java 内存数据库，号称"MySQL 兼容模式"。但有 3 个细节坑：

  1. SELECT ... FOR UPDATE 语义不同：H2 的 FOR UPDATE 在事务隔离级别上行为偏 PostgreSQL，跟 MySQL InnoDB 不一致
  2. utf8mb4 排序规则不同：H2 没有 utf8mb4_unicode_ci，写中文排序结果不一致
  3. AUTO_INCREMENT 行为略不同：插入失败后再插入，自增 ID 的跳跃行为有差异

学习项目用 H2 会埋"错误的肌肉记忆"。Testcontainers 启动慢 8 秒左右，但拿到真实 MySQL 8 语义，物超所值。

12.3 概念：Testcontainers 怎么工作 ​

@Testcontainers + @Container
   ↓
JUnit 启动测试前，Testcontainers 在 Docker 里拉起一个 MySQL 8.0 容器
   ↓
@DynamicPropertySource 把 datasource 的 url / username / password 改成指向容器
   ↓
Spring Boot 启动，Flyway 自动建表，跑测试
   ↓
测试结束，Testcontainers 关掉容器

12.4 编写基类 AbstractMySqlIT ​

路径：stock-service/src/test/java/com/snowvictor/cloudwork/stock/it/AbstractMySqlIT.java

java
package com.snowvictor.cloudwork.stock.it;

import org.springframework.boot.test.context.SpringBootTest;
import org.springframework.test.context.DynamicPropertyRegistry;
import org.springframework.test.context.DynamicPropertySource;
import org.testcontainers.containers.MySQLContainer;
import org.testcontainers.junit.jupiter.Container;
import org.testcontainers.junit.jupiter.Testcontainers;

/**
 * 集成测试基类: 启动一个 MySQL 8 容器, 把 datasource 指向它, 禁用 Nacos/Sentinel
 * 等真实组件以保证 IT 可离线运行。
 */
@SpringBootTest
@Testcontainers
public abstract class AbstractMySqlIT {

    @Container
    static final MySQLContainer<?> MYSQL = new MySQLContainer<>("mysql:8.0")
            .withDatabaseName("stock_db")
            .withUsername("stock_app")
            .withPassword("stock_app_pwd");

    @DynamicPropertySource
    static void overrideProps(DynamicPropertyRegistry r) {
        r.add("spring.datasource.url", MYSQL::getJdbcUrl);
        r.add("spring.datasource.username", MYSQL::getUsername);
        r.add("spring.datasource.password", MYSQL::getPassword);
        r.add("spring.cloud.nacos.discovery.enabled", () -> "false");
        r.add("spring.cloud.nacos.config.enabled", () -> "false");
        r.add("spring.cloud.nacos.config.import-check.enabled", () -> "false");
        r.add("spring.cloud.sentinel.enabled", () -> "false");
        r.add("spring.config.import", () -> "");   // override Nacos imports
    }
}

关键设计：

  • @SpringBootTest：启动完整 Spring 上下文（不是单元测试的 mock 环境）
  • @Testcontainers：JUnit 5 的扩展，配合 @Container 注解管理生命周期
  • MYSQL 是 static：所有继承 AbstractMySqlIT 的测试类共享同一个容器（每个测试类一个容器）。这是性能优化
  • @DynamicPropertySource：在 Spring Boot 启动前动态覆盖配置，让 datasource 指向容器；同时禁用 Nacos/Sentinel/config import，避免测试时连不上中间件

12.6 并发压测 IT（乐观锁版） ​

路径：stock-service/src/test/java/com/snowvictor/cloudwork/stock/it/StockDeductOptimisticIT.java

java
package com.snowvictor.cloudwork.stock.it;

import com.baomidou.mybatisplus.core.conditions.query.LambdaQueryWrapper;
import com.snowvictor.cloudwork.stock.domain.StockItem;
import com.snowvictor.cloudwork.stock.exception.OptimisticLockConflictException;
import com.snowvictor.cloudwork.stock.mapper.StockItemMapper;
import com.snowvictor.cloudwork.stock.service.StockService;
import com.snowvictor.cloudwork.stock.service.deduct.StockDeductStrategyResolver;
import org.junit.jupiter.api.BeforeEach;
import org.junit.jupiter.api.Test;
import org.springframework.beans.factory.annotation.Autowired;
import org.springframework.test.context.TestPropertySource;
import org.springframework.test.util.ReflectionTestUtils;

import java.util.concurrent.CountDownLatch;
import java.util.concurrent.ExecutorService;
import java.util.concurrent.Executors;
import java.util.concurrent.TimeUnit;
import java.util.concurrent.atomic.AtomicInteger;

import static org.assertj.core.api.Assertions.assertThat;

@TestPropertySource(properties = "stock.deduct.lock-mode=OPTIMISTIC")
class StockDeductOptimisticIT extends AbstractMySqlIT {

    @Autowired StockService stockService;
    @Autowired StockItemMapper itemMapper;
    @Autowired StockDeductStrategyResolver resolver;

    @BeforeEach
    void resetLockMode() {
        ReflectionTestUtils.setField(resolver, "mode", "OPTIMISTIC");
    }

    @Test
    void hundredThreads_eachDeductOne_zeroOversell() throws Exception {
        int threads = 100;
        ExecutorService pool = Executors.newFixedThreadPool(threads);
        CountDownLatch start = new CountDownLatch(1);
        CountDownLatch done = new CountDownLatch(threads);
        AtomicInteger success = new AtomicInteger();
        AtomicInteger conflict = new AtomicInteger();
        AtomicInteger insufficient = new AtomicInteger();

        for (int i = 0; i < threads; i++) {
            pool.submit(() -> {
                try {
                    start.await();
                    Integer r = stockService.deductStock("p1001", 1, "it-trace");
                    if (r != null && r >= 0) success.incrementAndGet();
                    else if (r != null && r < 0) insufficient.incrementAndGet();
                } catch (OptimisticLockConflictException e) {
                    conflict.incrementAndGet();
                } catch (Exception ignored) {
                } finally {
                    done.countDown();
                }
            });
        }
        start.countDown();
        boolean finished = done.await(30, TimeUnit.SECONDS);
        pool.shutdownNow();
        assertThat(finished).isTrue();

        StockItem after = itemMapper.selectOne(
                new LambdaQueryWrapper<StockItem>().eq(StockItem::getProductId, "p1001"));
        // 核心不变量: 成功扣减数 + 剩余 = 初始 100
        assertThat(success.get() + after.getStock()).isEqualTo(100);
        assertThat(after.getStock()).isGreaterThanOrEqualTo(0);   // 零超卖
        System.out.printf("OPTIMISTIC summary: success=%d conflict=%d insufficient=%d remain=%d%n",
                          success.get(), conflict.get(), insufficient.get(), after.getStock());
    }
}

讲解：

  • @TestPropertySource(properties = "stock.deduct.lock-mode=OPTIMISTIC")：测试启动时把锁模式设成 OPTIMISTIC
  • ReflectionTestUtils.setField：因为 StockDeductStrategyResolver 是 @RefreshScope 的，配置注入时机有点微妙，用反射强制设值最稳
  • CountDownLatch start：100 个线程都准备好后，主线程一声号令同时开干
  • 核心断言：success + remain == 100。如果出现超卖，这个等式会破坏（success > 100 或 remain < 0）

12.8 冲突审计 IT：证明 REQUIRES_NEW 生效 ​

这个测试是整个项目的精华——证明"主事务回滚后，独立事务的审计仍然存在"。

路径：stock-service/src/test/java/com/snowvictor/cloudwork/stock/it/StockDeductConflictAuditIT.java

java
@TestPropertySource(properties = "stock.deduct.lock-mode=OPTIMISTIC")
class StockDeductConflictAuditIT extends AbstractMySqlIT {

    @MockBean OptimisticDeductStrategy optimisticDeductStrategy;
    @Autowired StockService stockService;
    @Autowired StockDeductLogMapper logMapper;
    @Autowired StockItemMapper itemMapper;

    @Test
    void conflict_writesAuditEvenAfterRollback() {
        when(optimisticDeductStrategy.mode()).thenReturn(LockMode.OPTIMISTIC);
        when(optimisticDeductStrategy.deduct(anyString(), anyInt()))
                .thenThrow(new OptimisticLockConflictException("simulated"));

        StockItem before = itemMapper.selectOne(
                new LambdaQueryWrapper<StockItem>().eq(StockItem::getProductId, "p1003"));
        int beforeStock = before.getStock();

        assertThatThrownBy(() -> stockService.deductStock("p1003", 1, "conflict-trace"))
                .isInstanceOf(OptimisticLockConflictException.class);

        // 主事务回滚: 库存未变
        StockItem after = itemMapper.selectOne(
                new LambdaQueryWrapper<StockItem>().eq(StockItem::getProductId, "p1003"));
        assertThat(after.getStock()).isEqualTo(beforeStock);

        // 但失败流水通过 REQUIRES_NEW 已提交
        Long auditCount = logMapper.selectCount(new LambdaQueryWrapper<StockDeductLog>()
                .eq(StockDeductLog::getTraceId, "conflict-trace")
                .eq(StockDeductLog::getResultCode, 409));
        assertThat(auditCount).isEqualTo(1L);
    }
}

两个同时成立的断言：

  1. after.getStock() == beforeStock —— 主事务回滚，库存不变
  2. auditCount == 1 —— 但失败流水通过 REQUIRES_NEW 已提交

如果有一个断言失败，说明你的 REQUIRES_NEW 设计有问题（多半是 self-invocation 没避开）。

13.2 压测对比脚本 ​

跑完 IT 后，进入 stock_db 跑这段 SQL：

sql
SELECT lock_mode,
       COUNT(*)                                AS attempts,
       SUM(CASE WHEN result_code = 200 THEN 1 ELSE 0 END) AS ok,
       SUM(CASE WHEN result_code = 409 THEN 1 ELSE 0 END) AS conflicts,
       SUM(CASE WHEN result_code = -1  THEN 1 ELSE 0 END) AS insufficient,
       ROUND(AVG(cost_ms), 1)                  AS avg_ms,
       MAX(cost_ms)                            AS max_ms
FROM stock_deduct_log
GROUP BY lock_mode;

典型结果（数值会因机器而异）：

+--------------+----------+-----+-----------+--------------+--------+--------+
| lock_mode    | attempts | ok  | conflicts | insufficient | avg_ms | max_ms |
+--------------+----------+-----+-----------+--------------+--------+--------+
| OPTIMISTIC   |      100 |  56 |        38 |            6 |    3.2 |     12 |
| PESSIMISTIC  |      100 |  50 |         0 |           50 |    8.4 |     31 |
+--------------+----------+-----+-----------+--------------+--------+--------+

怎么解读：

  • 乐观锁：100 次尝试，56 次成功，38 次冲突失败，6 次因前面把库存扣完后才发现不足。整体 avg 3.2ms 快，但冲突多
  • 悲观锁：100 次尝试，50 次成功（恰好等于库存），0 次冲突，50 次因抢到锁后发现库存已空。整体 avg 8.4ms 慢，但确定性强

💡 从这张表能学到的最大一课：没有"最好的锁"，只有"适合场景的锁"。秒杀场景库存少冲突多，悲观锁更合适；普通业务冲突概率低，乐观锁更轻快。

第 14 章 · 踩坑清单与原因分析 ​

整理了本教程涉及的 10 个最容易掉的坑。每条都给"症状"+"根因"+"解法"。

14.1 self-invocation 让 @Transactional 失效 ​

  • 症状：明明加了 @Transactional，业务异常时事务没回滚，或 REQUIRES_NEW 跟主事务一起回滚
  • 根因：同类内方法 A 直接调方法 B（B 有 @Transactional），绕开了 Spring 代理
  • 解法：把 B 拎到独立 Bean，通过依赖注入调用

14.2 @Version 字段忘了加 NOT NULL DEFAULT 0 ​

  • 症状：第一次插入数据后，乐观锁更新失败，rows=0
  • 根因：NULL 跟任何东西比较都是 NULL，WHERE version = NULL 永远不命中
  • 解法：建表时 version INT NOT NULL DEFAULT 0，并在实体里给默认值

14.3 SELECT FOR UPDATE 没走索引锁全表 ​

  • 症状：高并发悲观锁场景，所有更新都串行，性能崩溃
  • 根因：WHERE 后的列没有索引，InnoDB 退化成表锁
  • 解法：把锁条件用的列加唯一索引（本教程的 uk_product_id）

14.4 HikariCP 的 max-lifetime 大于 MySQL wait_timeout ​

  • 症状：服务跑一段时间后偶发 CommunicationsException: The last packet successfully received...
  • 根因：MySQL 主动关闭空闲超时的连接，HikariCP 不知道，下次拿出来用就报错
  • 解法：让 max-lifetime 比 wait_timeout 小至少 30 秒

14.5 @Transactional 默认不回滚 checked exception ​

  • 症状：抛了 IOException 之类，事务没回滚，数据不一致
  • 根因：默认 rollbackFor = RuntimeException.class，checked exception 不算
  • 解法：始终写 @Transactional(rollbackFor = Exception.class)

14.6 Map<String, StockDeductStrategy> 用 bean name 做 key ​

  • 症状：策略解析返回 null，因为找不到 "OPTIMISTIC" 这个 key
  • 根因：Spring 注入 Map 时默认 key 是 bean name（如 "optimisticDeductStrategy"），不是 LockMode.name()
  • 解法：构造时自己用 strategy.mode().name() 重建 Map

14.7 Flyway 文件名错误导致迁移不执行 ​

  • 症状：表没建，应用启动报"table not found"
  • 根因：文件名漏写了双下划线（V1_xxx.sql 而不是 V1__xxx.sql），或者 V 没大写
  • 解法：严格遵守 V<版本>__<描述>.sql 命名

14.8 Nacos 改了配置但 @RefreshScope Bean 没刷新 ​

  • 症状：Nacos 改 lock-mode 后，下一次请求还用旧值
  • 根因：忘了在 Resolver 上加 @RefreshScope，或者 Nacos 配置 dataId 写错
  • 解法：Resolver 加注解；Nacos 后台确认配置生效（看日志 Refresh keys changed）

14.9 Testcontainers 启动太慢导致 CI 超时 ​

  • 症状：本地跑 IT 8 秒搞定，CI 上要 5 分钟
  • 根因：CI 没缓存 Docker 镜像，每次都下载 mysql:8（~200MB）
  • 解法：CI 配置加 Docker layer cache（GitHub Actions 的 docker/build-push-action 自带）

14.10 把 Feign 调用放进了 @Transactional ​

  • 症状：高并发时连接池告警 "Connection is not available, request timed out"
  • 根因：事务太长，连接被网络 IO 占用
  • 解法：把 Feign 调用提到事务外，本地事务只覆盖 DB 操作

第 15 章 · 下一步学什么 ​

跟完本教程，你已经掌握了"微服务 + MySQL 持久化"的标准实战路线。下一步可以按这个顺序学（详见设计文档 §11 Future Work）：

优先级 1：Redis 与分布式锁 ​

  • Redis 基础接入（Lettuce）：商品库存做 Cache-Aside，学穿透/击穿/雪崩三大缓存问题
  • Redisson 分布式锁：加入第三种锁策略 RedisDistributedLockStrategy，与现有两种锁做三方对比。我们在第 6 章的策略模式已经给它留好位置了——只需新增一个 @Component，零侵入

优先级 2：最终一致性 ​

  • 本地消息表 / Outbox 模式：解决本教程第 11.4 节"诚实声明"留下的跨服务一致性缺口
  • RabbitMQ 接入：由 Outbox 投递端可靠发送
  • 库存异步扣减：把同步 Feign 链路改成 RabbitMQ 异步事件流

优先级 3：可观测性（强烈建议提前学） ​

  • SkyWalking 链路追踪：把现有 traceId 升级为分布式 trace
  • Prometheus + Grafana：暴露 HikariCP 池指标、MyBatis 慢 SQL 指标

优先级 4：分布式事务 ​

  • Seata AT 模式：@GlobalTransactional 自动回滚跨服务调用
  • Seata TCC 模式：业务侵入式但更灵活

优先级 5：数据层进阶 ​

  • MySQL 主从 + 读写分离（ShardingSphere-JDBC）
  • 分库分表：按 user_id 取模 + Snowflake ID
  • Canal 监听 binlog：把 MySQL 变更同步到 Redis/ES

优先级 6：搜索 ​

  • ElasticSearch 商品搜索：异构存储协作

通用穿插 ​

  • 接口签名 + 防重放
  • 敏感字段加密
  • 慢 SQL 治理
